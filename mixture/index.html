Introduction

This page explains how you can configure a project and details some of the unique features Mixture has to offer.

Settings

Every Mixture project should contain a simple JSON file in the root called mixture.json.

{
  "debug": false,
  "lintScripts": false,
  "preprocessorLocations": null,
  "preprocessorScriptOutput": null,
  "preprocessorStyleOutput": null,
  "projectName": "Your project name",
  "projectDescription": "Your project description",
  "publishedId": null,
  "serverHostname": null,
  "serverPort": null,
  "templateDirectory": "templates",
  "templateDefaultLayout": "layout",
  "templateDefaultPage": "index",
  "type": "mixture",
  "useCompass": false,
  "simpleModePartnerServer": null,
  "modelSource": null,
  "domain": null,
  "followLeader": true,
  "customRuby": false,
  "customRubyPath": null,
  "routes": null,
  "remoteDebugPort": 0,
  "simpleModeFileTypes": null,
  "optimiseImages": true,
  "password": null,
  "ignoreDir": "ignore",
  "customTodos":  null,
  "autoPrefixer": null,
  "sass": null,
  "sitemap": false,
  "convertHtml": null,
  "less": null,
  "optimiseSVG": true,
  "reload": 0
}

Below is a breakdown of each element:

  • debug true or false, more information.
  • lintScripts true or false, tells Mixture to lint scripts when updated, more information.
  • preprocessorLocations sets the (folder) locations within your project that Mixture will handle preprocessing, minifcation & linting. eg "preprocessorLocations": ["public/styles", "public/scripts"]. if null no processing will happen.
  • preprocessorScriptOutput sets the output directory of any preprocessed scripts. If null they will be compiled in the same location as the source file.
  • preprocessorStyleOutput sets the output directory of any preprocessed styles. If null they will be compiled in the same location as the source file.
  • projectName name of the project.
  • projectDescription description of the project.
  • publishedId once published to the Mixture servers this will be populated with a unique project id.
  • serverHostname when set to null Mixture will attempt use your internal network address, if this fails it will fallback to "localhost" to use a custom hostname, please view our support site.
  • serverPort tells Mixture what port to use when running locally, if null Mixture will use an available port.
  • templateDirectory if you are building a Mixture project then all of your template files should be located within this folder, this folder requires two sub-directories: "layouts" and "includes" (these sub-directories can not be renamed).
  • templateDefaultLayout sets the default layout file for pages. This should be set to the name of your default layout within your "templates/layouts" folder without the .liquid extension. eg "templateDefaultLayout": "layout". You can stop all templates using a layout by setting to: "none", "" or null.
  • templateDefaultPage this tells mixture which template file to use as the root document (homepage). This should be set to the name of the page without the .liquid extension. eg "templateDefaultPage": "index"
  • type this tells Mixture what type of project it is and how to handle (currently only "mixture" and "simple" available).
  • useCompass if true tells Mixture not to handle Sass preprocessing and instead hands over to Compass. Mixture will still report issues.
  • simpleModePartnerServer if you are using Mixture in "simple" mode with another server (rails, mamp, etc) but still want to benefit from Mixture's multi-device and live refresh features, simple enter the address of the other server. eg "simpleModePartnerServer": "http://localhost:1234"
  • modelSource allows you to swap the local model system for online models.
  • domain add a custom domain to your online published project.
  • followLeader expects true or false, default is true. Enables & disables "follow the leader" functionality when using with multiple devices / browsers.
  • customRuby tells Mixture to use a custom version of Ruby, read more about custom Ruby.
  • customRubyPath allows you to override the path(s) to your own install of Ruby, read more about custom Ruby.
  • routes allows custom routes and redirects for your project read more.
  • remoteDebugPort set this port to enable remote debugging for devices, Mixture uses weinre – read more. When you open a project with this port set, mixture will alert you with the address you can use to start debugging with.
  • simpleModeFileTypes sets the file types that will cause a reload / refresh in simple mode. eg "simpleModeFileTypes": [".php", ".css"] — if this is not set (null) Mixture will reload all files.
  • optimiseImages expects true or false, when set to true Mixture will optimise images when added to the project using optipng and jpegtran.
  • password allows you to secure a published project with a password. To remove set back to null.
  • ignoreDir allows you to set an array of project directories that will be ignored by publish, convert to html and any preprocessing logic eg "ignoreDir": ["assets","public/css"].
  • customTodos please see the To-do section.
  • autoPrefixer please see the Autoprefix section.
  • sass please see the Sass section.
  • sitemap please see the Sitemap section.
  • convertHtml please see the convert Html section.
  • less please see the Less section.
  • optimiseSVG when true will optimise any SVGs added to the project while Mixture is running.
  • reload Delays or disables live reload and style injection. 0 = no delay. -1 = disabled. Anything above 0 and below 10000, will delay in milliseconds.

Live Refresh & Multi-device testing

Mixture will automatically reload or style inject when it detects changes in your project. This feature will work on most browsers and devices and is perfect for multi-device testing.

The Mixture live refresh feature also implements "follow the leader", this means any device / browser connected will change to reflect the one being navigated.

Mixture will automatically manage live reload on your behalf, you don't need to add any tags set anything up - just get busy building!

Mixture will also list all connected devices and if you set a "remoteDebugPort" in the Mixture settings, you can even click a device from Mixture to automatically open a WEINRE debug page for that device.

Changes to styles (stylesheets) are injected instead of a browser refresh.

Local Server

Mixture has a built-in local server for running Mixture projects.

When opening / starting a Mixture project, the server will be started on port 7774 if this is not possible an available port will be selected. Mixture will attempt to connect to your local network (this allows you easily to connect and test across devices on your local network), if you are not on a network or Mixture can not connect it will default to localhost.

You can override this behaviour and set your preffered hostname and port via the Mixture settings. If you are using the Windows version you may be prompted to confirm Mixture access to the port and hostsname.

If Mixture is unable to use the port set, it will always fallback to 7774 or any available port.

If you encounter issues starting the server from "view locally" or it is greyed out, please visit our support site and knowldege base.

Boilerplates

Boilerplates allow you to jump start a project quickly & easily. Some of the more popular boilerplates are already available in Mixture, but you can also create and share your own.

Please make sure your boilerplate fully works before you make public, we will test and remove if we find issues. It's also worth making sure that you do not have any custom settings that may not be replicated outside of your machine (custom ruby for example).

Unless explicitly detailed as to why it doesn't, please ensure that your boilerplate works when published to the Mixture servers - failing to do either of these may result in your boilerplate being switched to private.

The following is a brief explanation as to how you can add your own private or public boilerplates to Mixture.

  • Boilerplates are simply Mixture projects, the best way to get started is to create a new empty project folder, open with Mixture and select the "Empty" boilerplate.
  • Once downloaded and running, edit the project, settings and build your perfect starting point.
  • To add your boilerplate to Mixture you will first need to add your project as a Git repository and push it to GitHub (view more details about getting started with Git and GitHub).
  • Once your project is successfully in GitHub and you have a URL for this new repository, login to your online Mixture account.
  • Once signed in and if not already in the Boilerplates section, select Boilerplates from the main navigation.
  • You can now fill in some basic details and add your boilerplate as outlined, you can also manage your boilerplates from this area.

Smart To-dos

A helpful feature of Mixture is the ability to add to-dos in your code, to-dos are essentially comments, but they come in 3 flavours depending on what type of file or element you are adding them to...

<!-- TODO: I must not forget to close all of my html tags! -->

Can be used in the following file types: Liquid.

/* TODO: I must not forget to finish this project! */

Can be used in the following file types: CSS, Sass, Scss, Less, JavaScript or Liquid (when used in an inline style or script block).

These style comments used in preprocessors like Sass won't be picked up in the source file, but will be picked up in the output.

# TODO: I must not forget how many coffees I've drunk today! #

Can be used in the following file types: CoffeeScript.

-# TODO: I must check my indents, tabs or spaces?

Can be used in the following file types: Haml.

// TODO: preprocessing is so cool!

Can be used in the following file types: Sass, Scss, Less & Styl.

When Mixture detects a to-do it will automatically add and display the to-do within the application for easy access, clicking on a to-do within Mixture will open the file with the to-do within your default editor (for that file type).

Custom To-dos

You can create custom to-do logic and also define the file types that are affected

To do this you need to make an addition to the settings file, the following example shows you how to add support for to-do's in PHP, CS and Jade files.

"customTodos": [
	{
		"fileTypes": "php,cs",
		"regex": "(?<=/[*] TODO:).*?(?=[*]/)"
	},
	{
		"fileTypes": "jade",
		"regex": "(?<=//- TODO:).*?(?=$)"
	}
]
